## 1. 背景与目标

### 当前问题

- 单文件架构（app.py 1100+ 行），难以维护和扩展
- 无历史记录管理，分析完就找不到
- 每次都要重新配置研究背景和模型
- 问答是单轮，没有上下文
- 只能单文件上传，不能批量
- Streamlit 界面交互能力有限

### 目标

打造一个**真正可用、方便、好用**的科研资料整理工具，支持三种使用模式：

1. **单篇阅读**：快速出阅读卡片和写作素材
2. **批量处理**：监控文件夹，自动处理新文件
3. **课题积累**：围绕课题持续积累资料和关联分析

### 设计原则

1. **前后端分离**：通过 REST API 通信
2. **模块化**：Agent 逻辑独立成模块，可复用
3. **配置持久化**：用户配置保存到本地文件
4. **文件系统优先**：历史记录用文件系统 + 元数据索引

## 3. API 设计

### 核心接口

| 模块 | 接口 | 方法 | 说明 |
|------|------|------|------|
| 分析 | `/api/analysis/single` | POST | 单篇分析 |
| 分析 | `/api/analysis/batch` | POST | 批量分析 |
| 历史 | `/api/history` | GET | 获取历史列表（支持搜索/筛选） |
| 历史 | `/api/history/{id}` | GET | 获取单条详情 |
| 历史 | `/api/history/{id}` | DELETE | 删除记录 |
| 问答 | `/api/qa/chat` | POST | 多轮问答 |
| 问答 | `/api/qa/cross-doc` | POST | 跨文档问答 |
| 监控 | `/api/watch/start` | POST | 启动文件夹监控 |
| 监控 | `/api/watch/stop` | POST | 停止监控 |
| 监控 | `/api/watch/status` | GET | 监控状态 |
| 配置 | `/api/config` | GET | 获取配置 |
| 配置 | `/api/config` | PUT | 更新配置 |

### 数据流

**单篇分析**：
```
前端上传文件 → /api/analysis/single → 读取文件 → 分块 → 5个Agent依次执行 → 保存报告 → 返回结果
```

**批量处理**：
```
前端上传多文件 → /api/analysis/batch → 逐个处理 → 返回汇总
```

**文件夹监控**：
```
启动监控 → 检测新文件 → 自动触发分析 → 结果存入 history
```

## 4. Agent 模块设计

### 现有 Agent（从 app.py 提取）

| Agent | 输入 | 输出 | 改进点 |
|-------|------|------|--------|
| literature_parser | text_chunks | 阅读卡片 | 支持增量解析（大文件分批） |
| project_relation | 阅读卡片 + 研究背景 | 关联分析 | 支持多篇对比 |
| writing_agent | 阅读卡片 + 关联分析 + 写作风格 | 写作素材 | - |
| todo_agent | 阅读卡片 + 关联分析 | 任务清单 | 支持导出为可执行格式 |
| qa_agent | 文档 + 分析报告 + 问题 | 回答 | 多轮上下文 + 跨文档 |

### 新增能力

**多轮问答**：
- 维护 `conversation_id`，存储对话历史
- 每轮问答携带最近 N 轮上下文
- 支持追问、澄清、深入

**跨文档问答**：
- 输入：问题 + 多篇文档 ID
- 后端：从多篇文档中检索相关内容，拼接上下文
- 输出：回答 + 引用来源标注

**课题积累模式**：
- 为每个课题创建一个"课题空间"
- 课题空间下关联多篇文档的分析结果
- 支持课题维度的汇总和问答

## 5. 前端界面设计

### 设计方向

- **Tone**：温暖学术 + 现代科技感
- **配色**：深蓝灰底（#1a1a2e）+ 暖金点缀（#f0c040）+ 纸白内容区（#faf8f5）
- **字体**：标题用思源宋体（学术感），正文用思源黑体（清晰）
- **特色**：左侧导航 + 右侧工作区，工作流用时间轴可视化

### 页面结构

```
┌─────────────────────────────────────────────────────────┐
│  Note Forge                              用户头像 · 设置 │
├──────────┬──────────────────────────────────────────────┤
│          │                                              │
│  单篇分析 │   [当前页面内容区]                             │
│          │                                              │
│  批量处理 │   根据左侧选择切换：                            │
│          │   - 单篇：上传 → 配置 → 分析 → 结果            │
│  历史记录 │   - 批量：文件夹选择 → 监控状态 → 处理队列       │
│          │   - 历史：搜索/筛选 → 列表 → 详情              │
│  课题空间 │   - 课题：课题列表 → 文档关联 → 汇总问答         │
│          │   - 问答：对话界面                             │
│  智能问答 │                                              │
│          │                                              │
├──────────┴──────────────────────────────────────────────┤
│  状态栏：监控状态 · 处理进度 · 系统信息                     │
└─────────────────────────────────────────────────────────┘
```

### 关键交互

1. **单篇分析页**：拖拽上传 → 侧边栏配置 → 分析按钮 → 工作流时间轴动画 → 结果卡片切换
2. **批量处理页**：选择文件夹 → 监控开关 → 处理队列表格 → 实时进度
3. **历史记录页**：搜索框 + 筛选标签 → 卡片列表 → 点击查看详情
4. **课题空间页**：课题卡片 → 文档列表 → 汇总分析 + 问答入口
5. **问答页**：类似 ChatGPT 的对话界面，支持引用标注

### 文件结构

```

│   ├── 20260527_140756_paper1_record.json
```

### config.json 结构

```json
{
  "model": {
    "provider": "MiniMax",
```

### 历史记录索引

每条分析记录的 `record.json` 包含：
- `id`: 唯一标识
- `file_name`: 原始文件名
- `created_at`: 分析时间
- `model`: 使用的模型
- `topic_id`: 关联课题（可选）
- `tags`: 用户标签
- `summary`: 一句话摘要

## 7. 错误处理与边界情况

### 错误处理

| 场景 | 处理方式 |
|------|---------|
| LLM 调用失败 | 重试 2 次，仍失败则返回错误 + 已完成的部分结果 |
| 文件读取失败 | 跳过该文件，记录错误，继续处理其他文件 |
| 文件夹监控异常 | 停止监控，记录日志，前端提示 |
| 配置文件损坏 | 使用默认配置，提示用户重新设置 |
| 前后端通信断开 | 前端显示连接状态，自动重连 |

### 边界情况

- **大文件**：超过 50MB 的 PDF，提示用户确认后再处理
- **扫描版 PDF**：检测到无文本内容，提示需要 OCR
- **并发分析**：限制同时进行的分析任务数（默认 3）
- **磁盘空间**：定期检查，空间不足时提示清理
